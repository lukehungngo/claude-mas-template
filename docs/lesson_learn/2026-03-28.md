# Lessons Learned — 2026-03-28

## Session Context

Full-day session auditing 56 Claude Code sessions across 4 projects, cross-checking MAS pipeline compliance, and implementing architectural fixes. 8 commits, 4 review rounds, 2 differential reviews.

---

## Architectural Lessons

### 1. Orchestrator-as-subagent doesn't work

**What happened:** The Agent tool is not available to Level 1 subagents at runtime. The Orchestrator was dispatched once, reported "Agent tool not available," and the main session silently took over.

**Lesson:** Never depend on capabilities that haven't been empirically verified at the required nesting depth. If a tool works at Level 0 but fails at Level 1, redesign to use Level 0 dispatch.

**Fix:** Flat dispatch — dev-loop dispatches all agents directly. Orchestrator deprecated.

### 2. Enforcement paradox — more rules = less compliance

**What happened:** The system had 191 rules across 2,774 lines. The response to 0% compliance was adding more enforcement (checkpoint assertions, self-audit checklists, artifact gates). But the system's own meta-lesson says "prose instructions get skipped."

**Lesson:** When compliance is 0%, question the design before adding enforcement. A 5-step pipeline that's followed beats a 9-step pipeline with enforcement mechanisms.

**Fix:** Reduced to ~103 rules (46% cut). Removed duplicates, restatements, and agent-inherited rules. Each rule now exists in exactly one place.

### 3. The "Does NOT Do" anti-pattern

**What happened:** Every agent had non-negotiables at the top AND a "Does NOT Do" section at the bottom restating the same rules in negative form. 14 of 18 "Does NOT Do" items were pure restatements.

**Lesson:** Stating a rule positively once is better than stating it positively once and negatively once. The negative restatement adds noise without adding enforcement.

**Fix:** Stripped "Does NOT Do" sections to only keep role boundary items (things not obvious from non-negotiables).

### 4. Rules sections in commands restate their own steps

**What happened:** dev-loop.md had 15 rules at the bottom. 11 of them restated what Step 6, checkpoints, or gates already said inline. The rules section was a summary of the command — not new information.

**Lesson:** If a rule is already enforced by a gate or checkpoint at point of use, restating it in a separate "Rules" section is duplication. The model reads the gate when it reaches it — it doesn't need a table of contents.

**Fix:** Cut dev-loop Rules from 15 to 4. Cut bug-fix Rules from 6 to 1.

## Pipeline Lessons

### 5. Zero sessions completed the 9-step pipeline

**What happened:** Across 5 active MAS sessions, 0 completed all 9 steps. Steps 7 (Validate), 8 (Verify), and 9 (Finish) were skipped in 100% of sessions. The most productive session used only 2-3 steps.

**Lesson:** If zero users complete a workflow, the workflow has too many steps. Steps that are always skipped may be low-value, not under-enforced.

**Recommendation (not yet implemented):** Consider reducing to 6 steps: Clarify, Branch, Plan, Orchestrate, Review, Finish.

### 6. --auto was universally misinterpreted

**What happened:** In 4/4 --auto sessions, the flag was treated as "skip the entire pipeline" rather than "skip human approval gates." Only worktree creation survived.

**Lesson:** Ambiguous flag names get interpreted in the way that requires least effort. "Auto" without precise definition = "skip everything possible."

**Fix:** BAD/GOOD example pairs showing exactly what --auto skips (4 human pauses) vs what it doesn't skip (5 agent dispatches).

### 7. Bug-fixer agent was never dispatched

**What happened:** In 3/3 bug-fix sessions, the main session debugged and fixed code directly. Zero bug-fixer agent dispatches. The reproduction-test-first requirement was completely bypassed.

**Lesson:** Specialized agents exist to enforce discipline (reproduction test before fix). When the main session bypasses them, it also bypasses the discipline they enforce.

**Fix:** Checkpoint assertion + artifact gate in bug-fix command.

## Bootstrap & Plugin Lessons

### 8. Bootstrap was duplicating the plugin

**What happened:** Bootstrap copied all agents, skills, commands, and rules into `.claude/` locally. Users then had two copies of everything (plugin cache + local). These could drift apart.

**Lesson:** Don't duplicate what the plugin already provides. Bootstrap should only handle project-specific setup (CLAUDE.md, hooks, directories).

**Fix:** Lightweight bootstrap — only detects stack, fills CLAUDE.md, creates hooks and directories. All agents/skills/commands come from the plugin via `mas:` prefix.

### 9. Rules with placeholders don't belong in universal plugin rules

**What happened:** `rules/architecture.md` had `{{Invariant 1}}` placeholders. It was loaded every session as a rule but contained no actual rules — just a template.

**Lesson:** Plugin rules must be universal (applicable to every project). Project-specific templates belong in CLAUDE.md or are created by bootstrap.

**Fix:** Removed `rules/architecture.md` from plugin. CLAUDE.md already has the Architecture Invariants section.

## Process Lessons

### 10. Differential review caught what 4 review rounds missed

**What happened:** The adversarial differential review of Step 6 issued a REJECT verdict, identifying that the Orchestrator architecture was fundamentally broken. Four normal review rounds had only found surface issues (stale references, wrong counts).

**Lesson:** Normal reviews check correctness. Differential reviews check feasibility. Both are needed but serve different purposes. Differential review should be used for architectural decisions, not just research proposals.

### 11. Self-audit checklists are self-graded exams

**What happened:** The self-audit checklist asks the model to verify its own compliance. But the model that skipped steps is the same model filling in the checklist.

**Lesson:** Self-assessment has limited value. Artifact-based gates (file existence checks) are more reliable because they verify evidence, not self-reported compliance.

**Fix:** Added artifact verification gate (`ls docs/results/TASK-*-result.md`) as load-bearing enforcement alongside the self-audit.

### 12. Rule duplication grows silently

**What happened:** The system accumulated 191 rules across 12 files. Nobody noticed because each file seemed reasonable in isolation. The duplication only became visible when auditing across all files simultaneously.

**Lesson:** Rules need periodic cross-file audits. A rule that exists in 7 places isn't 7x more likely to be followed — it's 7x more likely to drift and confuse.

**Fix:** Single-source-of-truth principle. Each rule exists at the point of enforcement. Commands reference agent rules, agents reference severity-discipline.md, etc.
